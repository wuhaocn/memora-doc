## 2. 当前阶段总目标

把 Memora 从“单用户知识库 demo + 杂项模块堆叠”重构成一个有主链路的在线文档系统基础平台。

核心主链路：

`租户 -> 知识库 -> 文档树 -> 编辑 -> 版本 -> 同步 -> 权限`

阶段要求：

- 先做能用
- 再做能管
- 再做能同步
- 最后做能生产

---

## 3. 里程碑总览

| 里程碑 | 阶段目标 | 输出结果 |
|------|------|------|
| M0 | 方向收敛 | 文档体系、架构边界、模块定位统一 |
| M1 | 基础平台可运行 | 工作台、知识库、文档树、编辑、版本、同步任务、最小权限可运行 |
| M2 | 内容管理可闭环 | 知识库管理、文档新增编辑移动排序、权限界面可用 |
| M3 | 同步链路可闭环 | 客户端扫描、同步协议、增量同步初版完成 |
| M4 | 基础生产能力 | 权限、审计、搜索、发布、回收站完成基础版 |

当前状态：

- `M0` 已基本完成
- `M1` 已基本完成
- `M2` 已启动但未闭环
- `M3/M4` 尚未完成

## 5. P0 任务

P0 的目标是：

`把当前重构结果从“方向正确”变成“基础稳定”`

### 5.1 基础环境修复

#### 任务

- 修复 Gradle 本地运行问题
- 稳定前端跨架构依赖安装方式
- 明确本地开发环境要求
- 补充启动说明

#### 验收标准

- 后端可本地启动
- 前端可本地启动
- 前端可完成构建
- 后端可完成最小编译

### 5.2 认证上下文占位升级

当前进度：

- 已完成统一 `CurrentAccessContext`
- 当前优先通过 `Authorization: Bearer demo:{tenantId}:{userId}` 注入
- 已提供 `POST /api/v1/auth/login`
- 已提供 `GET /api/v1/auth/session`
- 旧的 `X-User-Id` / `X-Tenant-Id` 仍保留兼容
- 已移除核心在线文档主链路控制器中的硬编码默认用户写法

#### 任务

- 完全收敛所有模块到统一上下文策略
- 保持统一 `CurrentAccessContext`
- 保留过渡期兼容头部注入层
- 为后续真实鉴权接入预留替换点

#### 验收标准

- 业务接口不再直接写死用户
- 服务层通过统一上下文获取当前租户和当前用户
- Web 端通过统一会话上下文消费当前用户，而不是依赖整页刷新

### 5.3 基础测试

当前进度：

- 已补在线文档主链路最小集成测试骨架
- 已覆盖工作台、知识库创建、文档树、同步任务触发
- 已覆盖跨租户拒绝、租户角色拒绝、知识库级权限覆盖、知识库成员权限配置 API
- 当前后端最小集成测试已经可以稳定执行
- 已补主流程人工冒烟验收清单，明确 `admin / 123456` 的人工复验步骤
#### 任务

- 为工作台接口补最小集成测试
- 为知识库接口补最小集成测试
- 为文档树接口补最小集成测试
- 为同步任务触发补最小集成测试
- 为权限接口补最小集成测试

#### 验收标准

- 核心主链路接口具备最小回归保障

### 5.4 历史入口清理

#### 任务

- 清理首页上已不再属于主流程的模块入口
- 清理旧设计文档中的误导性主入口表述
- 标注资源库 / AI / 技能系统为“非当前主交付”

## 10. 风险清单

### 10.1 方向风险

- 如果再次把资源库 / AI / 技能系统拉回主链路，会重新打散重构重点

### 10.2 实现风险

- 如果先做编辑器而不做文档树管理，主流程仍然不闭环
- 如果先做自动同步而不做差异模型，会产生大量数据覆盖风险
- 如果先做分享而不做权限，会留下安全问题

### 10.3 组织风险

- 文档口径和代码口径若不一致，后续开发会重新回到旧架构

## 12. 验收建议

每一阶段验收都应该围绕主链路，而不是围绕模块数量。

正确的验收问题应该是：

- 能不能管理知识库
- 能不能管理文档树
- 能不能编辑文档
- 能不能看到版本
- 能不能把本地目录同步进来
- 能不能控制谁能看谁能改

如果这些问题回答不了，再多扩展模块都没有意义。
