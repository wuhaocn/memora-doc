# 开发规范

## 1. 目的

本文档定义 Memora 当前阶段的开发边界、改动原则、质量要求和协作约束。

当前规范只服务于这条主链路：

`tenant -> knowledge base -> document tree -> editing -> versions -> permissions`

旧 `DocStudio`、资源库、AI 技能系统、桌面同步客户端等历史叙事不属于当前基线。

---

## 2. 适用范围

适用于：

- `memora-server/`
- `memora-web-app/`
- `doc/`

适用于以下工作：

- 日常开发
- 缺陷修复
- 重构
- 提测前自检
- 文档更新

---

## 3. 基本原则

### 3.1 主链路优先

- 优先增强在线文档 Web 主流程，不优先扩外围系统。
- 如果一个改动不能明显改善主链路，应先说明收益再实施。
- 不为假设中的未来场景提前引入复杂抽象。

### 3.2 租户与权限边界显式

- 任何读取、写入、聚合逻辑都必须显式考虑租户边界。
- 任何知识库、文档相关接口都必须明确读、写或管理权限。
- 不能把前端隐藏按钮当成真正权限控制。

### 3.3 文档与实现一致

- README、架构文档、开发规范和运行行为应保持一致。
- 如果能力仍是占位或 demo 形态，必须直接写明。
- 不能把前端展示限制写成后端安全能力。

### 3.4 小步、可回归

- 一次改动解决一个清晰问题。
- 功能修改、重构修改、文档修改尽量解耦。
- 大改动要拆成能回归、能验收的小步。

---

## 4. 仓库结构约束

### 4.1 Backend

- `memora-server-common`：通用响应、异常、基础工具
- `memora-server-manager`：当前核心业务实现
- `memora-server-start`：启动、配置、数据库脚本、集成测试

约束：

- Controller 负责接口边界，不堆业务。
- Service 负责主业务逻辑和约束。
- Mapper 负责数据访问，不承载业务规则。
- DTO、VO、Entity 不混用职责。

### 4.2 Web

- `pages/`：页面容器和页面级状态
- `components/`：可复用 UI 与局部交互
- `services/api/`：接口封装
- `utils/`：纯工具逻辑，不堆业务主流程

约束：

- 页面组件不要无限膨胀；复杂页面应拆出子面板与局部状态。
- 按钮禁用态只是提示，不能替代服务端校验。
- 富文本、分享、权限相关 UI 必须明确区分展示能力与安全能力。
- 当前优先优化阅读、编辑、知识库和文档树操作流，而不是新增外围模块入口。

### 4.3 AI Context / Skills

如果项目接入 AI context、skills 或命令分发层，统一按下列结构执行：

- 唯一事实源：`ai-context/share-skills/`
- 项目级连接层：
  - `.claude/skills/`
  - `.codex/skills/`
  - `.cursor/skills/`
  - `.claude/commands/opsx/`
- 用户级投影层：
  - `~/.claude/skills/`
  - `~/.codex/skills/`
  - `~/.cursor/skills/`
  - `~/.claude/commands/opsx/`

约束：

- 共享技能内容只在 `ai-context/share-skills/` 维护。
- 项目级连接层只做连接、映射或镜像，不作为事实源直接修改。
- 用户级目录只是投影层，不承载长期 canonical 内容。
- 技能、命令、文档和脚本中不写绝对路径。
- 新增或调整技能时，必须检查项目级连接层和用户级投影层是否仍能正确解析。

## 5. 编码规范

### 5.1 通用要求

- 优先写直接、可读、可验证的实现。
- 不引入与当前阶段无关的基础设施。
- 修改前先理解当前主链路和 canonical 文档。
- 发现旧代码或旧文档与当前基线冲突时，优先修 canonical 文档。
- 涉及 AI context / skills 时，只修改事实源，不在多个投影层手工复制维护。

### 5.2 后端要求

- 一律通过 `CurrentAccessContext` 获取当前 tenant 和 user。
- 新增接口默认放在 `/api/v1/` 下，并使用资源语义命名。
- 预期业务失败使用 `BusinessException`，不要把业务失败抛成 500。
- 对唯一键、路径冲突、非法状态变更等做显式校验。
- 新增持久化字段时，同步更新：
  - `entity`
  - `schema.sql`
  - `data.sql`
  - 相关测试

### 5.3 前端要求

- 路由名、按钮文案、状态文案必须遵循当前产品语言。
- 组件使用 `PascalCase`，Hook 使用 `useXxx`。
- 页面级异步请求必须定义：
  - loading
  - empty
  - forbidden / not found
  - error
- 复制链接、阅读模式、权限提示等功能描述必须与真实权限能力分开。

### 5.4 数据与接口要求

- 文档树改动必须保证 `parentId`、`path`、`depth` 一致。
- 版本相关改动必须明确何时创建快照、何时允许回滚。
- 公开状态、分享、发布属于高风险能力，没有真实校验前不得写成已生产可用。

---

## 6. 安全红线

- 不允许把匿名访问当成已登录用户。
- 不允许只靠前端隐藏控件实现权限隔离。
- 不允许未经处理直接渲染不可信 HTML。
- 不允许把阅读链接等同于真实访问控制。

---

## 7. 测试与验证

### 7.1 最低要求

提交前至少完成与改动范围对应的验证：

- Backend：`./scripts/backend-test.sh`
- Web：`cd memora-web-app && npm run lint && npm run build`

### 7.2 当前环境说明

当前仓库已知本地环境限制：

- 后端统一通过仓库脚本调用 Gradle，默认优先使用 `./gradlew`
- 如果需要临时绕过 wrapper，可使用 `MEMORA_GRADLE_CMD=gradle` 覆盖脚本默认命令
- 使用 wrapper 时，脚本会默认把 `GRADLE_USER_HOME` 设到仓库内
- 后端测试在部分机器上仍可能因 Gradle native 兼容问题失败
- `./gradlew` 如果本地没有已缓存的 wrapper 分发包，还会依赖可访问的 Gradle 下载源
- Web 验证依赖本地 `node_modules`

如果因为环境问题无法完成验证，变更说明里必须写明：

- 哪个命令失败
- 为什么失败
- 已完成的替代验证

### 7.3 必做回归

涉及主链路改动时，至少回归：

- 登录与当前会话
- 工作台加载
- 知识库创建与进入
- 文档创建、编辑、阅读
- 版本列表与回滚
- 权限拒绝路径

人工回归以：

- [主流程冒烟验收清单](./online-document-main-flow-smoke-checklist.md)

为准。

## 8. 文档规范

- 根 `README.md` 负责仓库入口、当前范围和阅读顺序。
- 模块 `README.md` 负责模块职责、运行方式和当前状态。
- `doc/architecture/` 放产品、架构、设计与取舍文档。
- `doc/development/` 放开发规范、上线规范、验收清单和里程碑。
- AI context / skill 相关说明必须清楚区分：
  - 唯一事实源
  - 项目级连接层
  - 用户级投影层

更新规则：

- 系统行为变化时，优先更新对应 canonical 文档。
- 不新增“总结类”“完成报告类”平行文档替代正式文档。
- 历史计划或阶段记录如果保留，必须标明不是当前基线。
- 不把用户级投影目录写成 canonical 位置。

---

## 9. 代码评审要求

评审优先看：

- 是否偏离当前主链路
- 是否破坏租户隔离
- 是否绕过权限校验
- 是否引入前后端口径不一致
- 是否把占位能力写成生产能力
- 是否补齐必要验证

以下情况通常不通过：

- 未说明风险就放宽安全边界
- 大改动没有测试，也没有替代验证说明
- 重新引入 `DocStudio`、旧产品叙事或桌面客户端方向
- 文档、接口、页面文案明显互相矛盾

---

## 10. 变更说明要求

每次提交或合并请求至少说明：

- 改了什么
- 为什么改
- 影响范围
- 验证方式
- 未完成项或已知风险

如果涉及上线风险，还应补充：

- 回滚方式
- 数据兼容性说明
- 是否影响主流程验收清单
