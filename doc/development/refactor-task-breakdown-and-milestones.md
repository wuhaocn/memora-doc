## 2. 当前阶段总目标

把 Memora 从“单用户知识库 demo + 杂项模块堆叠”重构成一个有明确主链路的在线文档平台基础版。

核心主链路：

`tenant -> knowledge base -> document tree -> editing -> versions -> permissions`

阶段策略：

- 先做能用
- 再做能管
- 再做能好用
- 最后做能生产

---

## 3. 里程碑总览

| 里程碑 | 阶段目标 | 输出结果 |
| --- | --- | --- |
| M0 | 方向收敛 | 文档体系、架构边界、模块定位统一 |
| M1 | 基础平台可运行 | 工作台、知识库、文档树、编辑、版本、最小权限可运行 |
| M2 | 内容管理可闭环 | 知识库管理、文档新增编辑移动排序、权限交互可用 |
| M3 | Web 产品体验闭环 | 搜索、阅读体验、版本体验、权限交互继续收敛 |
| M4 | 基础生产能力 | 权限、审计、发布、回收站达到基础能力 |

当前状态：

- `M0` 已完成
- `M1` 基本完成
- `M2` 已启动但未闭环
- `M3/M4` 尚未完成

## 5. P0 任务

P0 目标：

`把当前重构结果从“方向正确”推进到“基础稳定”`

### 5.1 基础环境修复

任务：

- 修复本地 Gradle 运行问题
- 稳定前端跨环境依赖安装方式
- 明确本地开发环境要求
- 完善启动说明

验收标准：

- 后端可本地启动
- 前端可本地启动
- 前端可完成构建
- 后端可完成最小编译

### 5.2 认证上下文升级

当前进度：

- 已统一 `CurrentAccessContext`
- 当前通过 `Authorization: Bearer demo:{tenantId}:{userId}` 注入上下文
- 已提供 `POST /api/v1/auth/login`
- 已提供 `GET /api/v1/auth/session`
- 已移除主链路控制器中的硬编码默认用户和匿名兜底身份

任务：

- 把相关模块完全收敛到统一上下文策略
- 保持 `CurrentAccessContext` 为唯一访问入口
- 为后续真实认证保留清晰替换点

验收标准：

- 业务接口不再硬编码用户
- 服务层通过统一上下文获取当前租户和当前用户
- Web 端通过共享会话上下文消费当前用户，而不是依赖整页刷新

### 5.3 基础测试

当前进度：

- 已具备在线文档主链路最小集成测试骨架
- 已覆盖工作台、知识库创建、文档树
- 已覆盖跨租户拒绝、角色拒绝、知识库级权限覆盖、知识库成员权限接口
- 后端启动、验证、构建入口已统一到仓库脚本
- 测试用例存在，但 `./scripts/backend-test.sh` 仍可能受 Gradle native 环境问题阻塞
- `./gradlew` 在未缓存分发包时还受网络下载条件限制
- 已补主流程人工冒烟验收清单，明确 `admin / 123456` 的验证路径

任务：

- 为工作台接口补最小集成测试
- 为知识库接口补最小集成测试
- 为文档树接口补最小集成测试
- 为权限接口补最小集成测试

验收标准：

- 核心主链路接口具备最小回归保障

### 5.4 历史入口清理

任务：

- 清理首页上已不属于主流程的模块入口
- 清理旧设计文档中的误导性主入口表述
- 标注资源库 / AI / 技能系统 / 桌面客户端为“非当前主交付”

验收标准：

- 仓库、首页、文档三处口径一致

## 9. 风险清单

### 9.1 方向风险

- 如果再次把资源库 / AI / 技能系统 / 桌面客户端拉回主链路，重构重点会再次被打散

### 9.2 实现风险

- 如果先做编辑器而不补文档树管理，主流程仍然不闭环
- 如果先做分享而不做权限，会留下明显安全问题

### 9.3 组织风险

- 文档口径和代码口径如果再次分离，团队会重新滑回旧架构叙事

第一组：

- 修环境

第二组：

第三组：

- 搜索入口与结果页

- 审计与回收站

## 11. 验收建议

每一阶段验收都应围绕主链路，而不是围绕新增模块数量。

正确的验收问题应该是：

- 能不能管理知识库
- 能不能管理文档树
- 能不能编辑文档
- 能不能看到版本
- 能不能控制谁能看、谁能改
- Web 端是否已经具备连续、清晰、低摩擦的主操作流

如果这些问题答不清楚，再多外围模块也没有产品意义。
