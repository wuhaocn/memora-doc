# 上线规范

## 1. 目的

本文档定义 Memora 当前阶段的发布门槛、上线步骤、验收要求和回滚约束。

当前上线规范只覆盖在线文档 Web 主链路，不代表未来公开发布、完整协作或 AI 扩展已经具备生产能力。

---

## 2. 当前发布分级

### 2.1 本地开发验证

适用场景：

- 开发自测
- 单机联调

要求：

- 关键模块能本地启动
- 改动点完成最小验证

### 2.2 内部演示 / 联调环境

适用场景：

- 团队联调
- 内部试用
- 产品演示

要求：

- 主链路可走通
- 冒烟清单通过
- 已知问题已记录

### 2.3 受控试运行

适用场景：

- 小范围真实用户验证
- 固定租户试用

额外要求：

- 无匿名默认登录
- 权限拒绝路径可验证
- 富文本渲染风险已收敛到可接受基线
- 阅读链接行为与真实权限模型一致
- 至少具备基础监控和回滚方案

### 2.4 正式生产上线

当前状态：

- 本仓库当前仍不满足正式生产上线标准

主要原因：

- 真实认证未接入
- 受控阅读与发布未生产化
- 构建环境稳定性仍需收口

---

## 3. 上线前置门槛

### 3.1 变更范围明确

上线前必须说明：

- 本次上线解决什么问题
- 涉及哪些模块
- 是否影响主链路
- 是否涉及数据库、权限、阅读链接 / 发布、版本逻辑

### 3.2 自动化验证

上线前应完成：

- Backend：`./scripts/backend-test.sh`
- Web：`cd memora-web-app && npm run lint && npm run build`

如果某项因环境问题未执行，必须在上线说明中明确标注，并给出替代验证结果。

当前已知补充说明：

- Backend 在部分机器上可能同时受到 Gradle native 库兼容问题和 wrapper 下载条件限制
- 如需临时绕过 wrapper，可使用 `MEMORA_GRADLE_CMD=gradle` 调用仓库脚本
- 发布构建统一使用 `./scripts/release-build.sh`，它会先执行 `./scripts/backend-build.sh` 再构建 Web 产物
- 如果 `./scripts/backend-test.sh` 未通过，至少需要补充 `git diff --check`、Web 构建验证和主流程人工冒烟结果

### 3.3 人工冒烟

上线前至少按以下文档完成主流程人工回归：

- [主流程冒烟验收清单](./online-document-main-flow-smoke-checklist.md)

至少确认：

- 登录
- 工作台加载
- 创建知识库
- 创建文档
- 编辑保存
- 查看版本
- 复制阅读链接
- 退出登录

### 3.4 高风险变更额外评审

以下变更必须额外评审：

- 租户上下文与认证逻辑
- 权限判定逻辑
- 文档树移动、排序、删除
- 版本快照与回滚
- 阅读链接、公开状态、发布能力
- 数据库结构变更
- AI context / skills 的事实源、连接层或投影层结构调整

---

## 4. 上线清单

### 4.1 代码冻结前

- 确认需求范围收口
- 确认没有夹带无关重构
- 确认文档已同步更新

### 4.2 合并前

- 完成代码评审
- 完成自动化验证
- 完成人工冒烟
- 写明已知风险和未完成项

### 4.3 发布前

- 确认目标环境配置
- 确认数据库脚本和版本兼容性
- 确认前后端构建产物一致
- 优先通过 `./scripts/release-build.sh` 生成统一发布目录
- 确认回滚方式可执行
- 如果项目接入 `ai-context/share-skills/`，确认：
  - 事实源目录是唯一修改来源
  - 项目级连接层解析正常
  - 用户级投影层同步策略明确
  - 没有遗留绝对路径

### 4.4 发布后

- 回归核心主链路
- 检查关键日志和异常
- 检查权限拒绝行为
- 检查版本相关接口是否异常增长或失败

---

## 5. 环境要求

### 5.1 Backend

- Java 21
- 可用的 Gradle 环境，优先通过仓库脚本调用 wrapper
- H2 用于开发验证；切到 MySQL 时需补环境差异校验

### 5.2 Web

- Node.js 已安装且本地依赖可正常安装
- `npm run build` 可稳定执行

---

## 6. 数据与兼容性要求

### 6.1 数据库变更

涉及 `schema.sql`、实体字段或索引时，必须确认：

- 新字段是否有默认值或兼容路径
- 唯一约束是否会冲击现有数据
- 路径、slug、状态字段是否需要回填策略

### 6.2 接口兼容

涉及接口改动时，必须确认：

- Web 当前调用是否同步更新
- 返回结构是否破坏现有页面
- 权限失败是否仍返回明确业务错误

### 6.3 文档模型一致性

涉及文档模型改动时，必须确认：

- `parentId`
- `path`
- `depth`
- `versionNo`

不会进入不一致状态。

---

## 7. 安全与权限门槛

如果以下任一项不满足，不应进入受控试运行以上环境：

- 未认证请求仍能获取有效会话
- 服务端只靠前端按钮禁用阻止写操作
- 不可信 HTML 可直接渲染
- 阅读链接被错误描述为放宽权限

---

## 8. 上线说明模板

每次上线说明至少包含：

### 8.1 变更摘要

- 本次上线内容
- 明确不包含的内容

### 8.2 影响范围

- Backend / Web / Docs
- 是否影响主链路

### 8.3 验证结果

- 自动化验证结果
- 人工冒烟结果
- 未执行项及原因

### 8.4 风险说明

- 已知问题
- 临时兜底策略
- 需要重点观察的日志或接口

### 8.5 回滚方案

- 如何回滚应用代码
- 如何回滚前端构建产物
- 是否需要回滚数据库

---

## 9. 回滚规范

以下任一情况出现时，应立即评估回滚：

- 登录或会话获取异常
- 工作台无法打开
- 知识库创建或进入失败
- 文档编辑保存失败
- 版本回滚异常
- 权限越权或误拒绝

回滚要求：

- 先回滚影响主链路的应用版本
- 再确认数据库是否需要恢复
- 回滚后立即重走主流程冒烟
- 记录故障时间、影响范围、回滚结果

---

## 10. 当前基线结论

按当前仓库状态：

- 可以做本地开发验证
- 可以做内部演示 / 联调发布
- 不应宣称具备正式生产上线能力

如果要推进到受控试运行，至少先补：

- 真实认证上下文
- 受控阅读与发布控制
- 富文本安全收口
- 更稳定的构建与回归机制
