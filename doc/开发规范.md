# DocStudio 开发规范

## 1. 概述

本文档定义了 DocStudio 项目的开发规范，包括架构设计原则、代码组织规范、反思能力设计、自我评估机制和迭代优化闭环。遵循这些规范将有助于提高代码质量、系统可维护性和团队协作效率。

## 2. 架构设计原则

### 2.1 高内聚、低耦合

#### 2.1.1 高内聚原则

- **功能内聚**：每个模块、类或函数应专注于单一功能领域
- **信息内聚**：相关数据和操作应封装在同一模块中
- **通信内聚**：模块之间通过明确的接口通信，避免直接依赖内部实现
- **顺序内聚**：确保模块内的操作按逻辑顺序执行

#### 2.1.2 低耦合原则

- **接口隔离**：使用抽象接口而非具体实现进行通信
- **依赖倒置**：高层模块不应依赖低层模块，两者都应依赖抽象
- **最小知识原则**：一个模块只应了解与其直接相关的其他模块
- **无环依赖**：避免模块之间形成循环依赖
- **单一职责**：每个类只负责一个特定功能

### 2.2 分层架构

采用经典的分层架构，确保各层职责清晰：

```
┌─────────────────────────┐
│       表现层 (UI)        │
├─────────────────────────┤
│       业务层 (Service)   │
├─────────────────────────┤
│       数据访问层 (DAO)   │
├─────────────────────────┤
│       数据存储层         │
└─────────────────────────┘
```

### 2.3 模块化设计

- 按功能领域划分模块
- 每个模块应具有清晰的边界和接口
- 模块之间通过明确的协议通信
- 支持模块的独立开发、测试和部署

## 3. 代码组织规范

### 3.1 目录结构

#### 3.1.1 后端目录结构

```
memora-server/
├── memora-server-common/       # 公共模块
│   └── src/main/java/com/memora/common/
│       ├── exception/          # 异常定义
│       └── result/             # 统一响应
├── memora-server-manager/      # 管理模块
│   └── src/main/java/com/memora/manager/
│       ├── controller/         # 控制器
│       ├── dto/                # 数据传输对象
│       ├── entity/             # 实体类
│       ├── mapper/             # 数据访问层
│       ├── service/            # 服务层
│       └── vo/                 # 视图对象
└── memora-server-start/        # 启动模块
    └── src/main/java/com/memora/
        ├── config/             # 全局配置
        └── DocStudioApplication.java  # 启动类
```

#### 3.1.2 前端目录结构

```
memora-web-app/
├── src/
│   ├── components/          # 通用组件
│   ├── pages/               # 页面组件
│   ├── extensions/          # 扩展功能
│   ├── utils/               # 工具函数
│   ├── main.jsx             # 入口文件
│   └── App.jsx              # 根组件
└── vite.config.js           # Vite配置文件
```

### 3.2 命名规范

#### 3.2.1 后端命名规范

- **包名**：小写字母，使用点分隔，如 `com.memora.manager.controller`
- **类名**：大驼峰命名法，如 `KnowledgeBaseController`
- **方法名**：小驼峰命名法，动词开头，如 `createKnowledgeBase`
- **变量名**：小驼峰命名法，如 `knowledgeBaseId`
- **常量名**：全大写，下划线分隔，如 `MAX_PAGE_SIZE`

#### 3.2.2 前端命名规范

- **组件名**：大驼峰命名法，如 `DocumentEdit`
- **函数名**：小驼峰命名法，动词开头，如 `handleSaveDocument`
- **变量名**：小驼峰命名法，如 `isLoading`
- **文件名**：大驼峰命名法（组件）或小写字母+连字符（其他）

## 5. 自我评估机制

### 5.1 核心概念

自我评估机制是指系统能够定期评估自身状态和性能，并生成评估报告的能力。

### 5.2 实现方式

#### 5.2.1 代码质量评估

- **代码复杂度分析**：评估代码的圈复杂度、耦合度等指标
- **代码风格检查**：确保代码符合项目风格规范
- **测试覆盖率分析**：监控单元测试和集成测试的覆盖率

#### 5.2.2 系统性能评估

- **响应时间监控**：测量关键 API 的响应时间
- **资源利用率监控**：监控 CPU、内存、磁盘等资源的使用情况
- **吞吐量分析**：评估系统处理请求的能力

#### 5.2.3 业务指标评估

- **功能完成度**：评估迭代计划中功能的完成情况
- **用户满意度**：收集和分析用户满意度数据
- **系统稳定性**：监控系统的可用性和错误率

## 6. 迭代优化闭环

### 6.1 核心概念

迭代优化闭环是指通过持续的反馈、评估和改进，不断提高系统质量的过程。

### 6.2 实现方式

#### 6.2.1 迭代流程

1. **需求分析**：明确迭代目标和需求
2. **设计实现**：按照架构设计原则实现功能
3. **测试验证**：进行单元测试、集成测试和系统测试
4. **部署上线**：将新功能部署到测试环境
5. **监控评估**：收集系统运行数据，进行自我评估
6. **反思改进**：分析评估结果，提出改进建议
7. **优化迭代**：基于改进建议进行下一轮迭代

#### 6.2.2 工具支持

- **CI/CD 流水线**：自动化构建、测试和部署流程
- **监控系统**：实时监控系统运行状态
- **代码分析工具**：自动检查代码质量
- **评估报告生成工具**：自动生成系统评估报告

## 7. 技能开发规范

### 7.1 技能设计原则

- **单一职责**：每个技能专注于一个特定功能领域
- **模块化设计**：技能应采用模块化结构，便于维护和扩展
- **清晰的输入输出**：明确定义技能的输入参数和输出格式
- **良好的错误处理**：实现完善的错误处理机制
- **可测试性**：技能应易于测试，便于验证功能正确性
- **文档完整性**：提供详细的使用文档和示例

### 7.2 技能目录结构

```
skill-name/
├── SKILL.md              # 技能元数据和使用说明
├── manifest.json          # 技能清单文件
└── src/                   # 技能实现代码
```

### 7.3 技能开发流程

1. **问题定义**：明确技能要解决的问题和目标用户
2. **设计阶段**：定义输入输出，设计工作流程
3. **开发实现**：按照规范实现技能核心逻辑
4. **测试验证**：编写测试用例，验证功能正确性
5. **文档编写**：创建详细的使用文档和示例
6. **发布部署**：将技能部署到项目中
7. **维护更新**：收集反馈，持续改进技能

## 8. 代码审查规范

### 8.1 审查流程

1. **提交前自检**：开发者在提交代码前进行自我审查
2. **代码提交**：将代码提交到版本控制系统
3. **审查请求**：创建代码审查请求，指定审查人员
4. **代码审查**：审查人员检查代码质量、架构符合性等
5. **反馈修改**：开发者根据审查意见修改代码
6. **审查通过**：代码审查通过后合并到主分支

### 8.2 审查重点

- **架构符合性**：代码是否符合高内聚、低耦合原则
- **代码质量**：代码是否清晰、简洁、可维护
- **安全性**：代码是否存在安全漏洞
- **性能**：代码是否存在性能问题
- **测试覆盖**：是否有足够的测试用例

## 9. 测试规范

### 9.1 测试类型

- **单元测试**：测试单个函数或类的功能
- **集成测试**：测试模块之间的交互
- **系统测试**：测试整个系统的功能
- **性能测试**：测试系统的性能指标
- **安全测试**：测试系统的安全性

### 9.2 测试覆盖率

- **单元测试覆盖率**：目标 ≥ 80%
- **关键功能测试覆盖率**：目标 100%
- **集成测试覆盖率**：目标 ≥ 70%

## 10. 文档规范

### 10.1 代码文档

- **类注释**：描述类的功能、职责和使用方法
- **方法注释**：描述方法的功能、参数、返回值和异常
- **属性注释**：描述属性的含义和用途
- **复杂逻辑注释**：解释复杂算法或业务逻辑

### 10.2 项目文档

- **需求文档**：详细描述系统需求和功能规格
- **架构文档**：描述系统架构设计和组件关系
- **API文档**：详细描述API接口的使用方法
- **部署文档**：描述系统部署和配置方法
- **用户手册**：描述系统的使用方法

## 11. 开发触发规则

### 11.1 核心概念

开发触发规则是指在开发过程中可选择执行的检查和优化机制，旨在确保代码质量、架构一致性和持续改进。根据用户要求，自动检查已改为可选执行，默认不执行。

### 11.2 触发时机

- **代码提交前**：通过 Git 预提交钩子可选执行（默认跳过）
- **代码合并前**：通过 CI/CD 流水线执行（后续实现）
- **定期执行**：通过定时任务执行全面评估（后续实现）

### 11.3 触发流程

1. **代码反思阶段**
   - 检查代码冲突
   - 分析代码质量
   - 识别潜在问题

2. **自我评估阶段**
   - 编译检查
   - Lint 检查
   - 测试覆盖率检查
   - 性能评估

3. **迭代优化阶段**
   - 更新技能统计
   - 优化代码结构
   - 改进开发流程
   - 更新文档

### 11.4 实现机制

#### 11.4.1 Git 预提交钩子

Git 预提交钩子将在后续实现，当前默认直接提交代码。

#### 11.4.2 CI/CD 流水线集成

在后续开发中，我们将在 CI/CD 流水线中集成以下检查：

- **单元测试**：确保核心功能正常工作
- **集成测试**：验证模块间的交互
- **系统测试**：测试整个系统的功能
- **性能测试**：评估系统性能指标
- **安全测试**：检查潜在的安全漏洞

### 11.5 触发规则配置

| 触发时机 | 执行检查 | 失败处理 |
|---------|---------|---------|
| 代码提交前 | 冲突检查、编译检查、Lint 检查（可选） | 阻止提交，显示详细错误信息 |
| 代码合并前 | 所有提交前检查 + 单元测试、集成测试 | 阻止合并，生成详细报告 |
| 定期执行 | 全面评估 + 性能测试 + 安全测试 | 生成评估报告，通知相关人员 |

### 11.6 开发建议

- 定期更新开发规范文档
- 积极参与代码审查
- 关注技能使用反馈
- 持续改进开发流程
- 对于重要提交，建议手动执行检查：`DOCSTUDIO_CHECK=1 git commit`
