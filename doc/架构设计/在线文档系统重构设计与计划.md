# Memora 在线文档系统重构设计与计划

## 1. 文档目的

本文档用于说明当前重构后的系统目标、功能范围、设计方案、当前实现状态以及后续迭代计划。

这份文档聚焦当前实际可落地的版本，不再沿用此前“资源库 / AI / 技能市场 / 协作平台”全部并行推进的发散式规划，而是先收敛为一个可运营的在线文档系统基础平台。

---

## 2. 当前目标

### 2.1 产品目标

构建一个面向企业知识管理场景的在线文档系统，优先满足以下核心诉求：

- 多租户隔离
- 知识库管理
- 文档树管理
- 文档版本管理
- 本地文件同步
- 在线统一发布
- 后续可扩展协作、搜索、权限和 AI 能力

### 2.2 当前阶段的设计原则

- 先做基础平台，不做功能堆砌
- 先做租户、知识库、文档树、同步任务四个主骨架
- 保留后续扩展空间，但不提前引入复杂度
- Web 端先承担管理台角色
- 本地客户端后续承担同步代理角色

---

## 3. 当前系统定位

当前系统定位为：

`多租户在线文档系统 + 知识库平台 + 本地文件同步入口`

不是以下定位：

- 不是资源库优先的平台
- 不是 AI 功能展示站
- 不是一开始就做全量协作套件
- 不是通用 CMS

## 5. 当前设计方案

### 5.1 领域模型

当前核心模型如下：

- `Tenant`：租户 / 工作区
- `TenantMember`：租户成员
- `KnowledgeBase`：知识库
- `Document`：文档节点，支持 `DOC` 和 `FOLDER`
- `DocumentVersion`：文档版本快照
- `SyncJob`：同步任务

### 5.2 核心关系

- 一个租户包含多个知识库
- 一个租户包含多个成员
- 一个知识库包含多个文档节点
- 一个文档节点可以是目录或文档
- 一个文档可拥有多个版本
- 一个知识库可产生多条同步任务记录

### 5.3 文档树建模方式

文档树当前采用“父子关系 + 路径”的混合模型：

- `parentId`：表达直接父子关系
- `path`：表达完整树路径
- `depth`：表达层级深度
- `docType`：区分目录和正文文档

这样做的原因：

- 便于前端直接构建树
- 便于后端做路径更新
- 便于未来做路径检索和同步映射

### 5.4 本地同步建模方式

当前本地同步不是直接把桌面文件系统暴露成线上文档，而是采用“知识库绑定目录 + 同步任务记录”的方式：

- 知识库保存 `localRootPath`
- 文档保存 `sourceType`、`sourcePath`
- 同步任务保存扫描结果和变更数量
- 线上知识库仍然是统一发布面

设计意图：

- 避免本地目录直接成为线上真实源头而失控
- 保留线上修订、审核、版本管理能力
- 为后续冲突处理打基础

### 6.3 当前是否满足继续推进

结论：

- 已满足继续收口在线文档主链路
- 尚不满足继续扩展资源库 / AI / 实时协作等外围大模块

原因：

- 多租户、多知识库、文档树、基础编辑、版本能力已经成型
- 权限已形成最小闭环，但仍未完成生产级认证和更完整的可视化配置界面
- 同步仍只有任务骨架，尚未形成真实本地同步链路

## 7. 当前问题清单

### 7.1 架构层问题

- 旧文档和旧代码中仍保留资源库 / AI / 技能系统的主入口叙事，容易干扰产品聚焦
- 当前后端仍然是单体服务，模块边界刚刚清晰，未到拆服务阶段
- 业务接口仍偏 CRUD，缺少真正的领域动作接口

### 7.2 业务层问题

- 当前权限已细化到知识库级，并已具备最小可视化配置入口，但体验和生产化程度仍不足
- 当前鉴权仍是 demo token 占位，不是真实登录和组织身份体系
- 还没有知识库分享、发布、审核流程
- 还没有文档状态流转
- 还没有全文检索
- 还没有回收站与归档策略

### 7.3 客户端问题

- Tauri 客户端已收敛为同步代理最小壳，但还不是真实同步客户端
- 缺少本地扫描索引
- 缺少文件哈希与变更跟踪
- 缺少同步队列和错误恢复

## 8. 建议的目标架构

### 8.1 第一阶段

目标：把系统做成“能管理、能浏览、能编辑、能控权”的在线文档基础平台

范围：

- 多租户工作台
- 知识库管理
- 文档树管理
- 在线文档浏览与基础编辑
- 版本快照
- 最小权限控制

### 8.2 第二阶段

目标：补齐“能同步、能搜索、能发布”

范围：

- Tauri 同步代理
- 扫描与增量同步
- 冲突识别
- 搜索
- 知识库分享与发布
- 回收站与归档
- 文档操作审计

### 8.3 第三阶段

目标：补齐“真正可生产化的认证与协作能力”

范围：

- 真实认证与会话
- 更完整权限体系
- 更成熟编辑体验
- 协作能力扩展
- 自动同步与定时同步

## 10. 结论

当前系统已经从“单用户知识库 demo”转向“多租户在线文档平台骨架”，方向是正确的，但现在仍处在基础平台搭建阶段。

当前最重要的不是继续加新模块，而是把以下主链路做扎实：

- 租户
- 知识库
- 文档树
- 编辑
- 同步
- 权限

只有这条主链路稳定后，资源库、协作、AI、技能等能力才值得重新接入。
