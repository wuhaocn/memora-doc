# Memora 开源在线文档方案借鉴分析

## 1. 文档目的

本文档用于分析 GitHub 上已有的在线文档 / Wiki / 知识库系统，判断哪些设计思路适合 Memora 当前的在线文档模块，哪些能力值得借鉴，哪些实现不适合直接照搬。

重点不是“找一个项目照抄”，而是：

- 借鉴成熟产品的结构设计
- 借鉴领域模型和交互策略
- 避免重复踩坑
- 避免引入不必要的复杂度
- 注意开源许可证边界

---

## 2. 结论先行

有，而且不少。

但更准确地说：

- **可以大量借鉴设计思路**
- **可以参考模块划分和产品流程**
- **不建议直接拼装式抄代码**
- **必须先看许可证，再决定是否能复用实现**

如果按 Memora 当前阶段的目标来选，最值得重点研究的是五类项目：

- Outline
- Docmost
- BookStack
- Wiki.js
- AppFlowy / AFFiNE

其中：

- `Outline` 最适合借鉴“团队知识库 + 权限 + 发布”的产品结构
- `Docmost` 最适合借鉴“Workspace / Space / Page”层级和企业协作边界
- `BookStack` 最适合借鉴“内容层级清晰、低认知负担”的组织方式
- `Wiki.js` 最适合借鉴“存储抽象、编辑器抽象、搜索抽象”的平台化设计
- `AppFlowy / AFFiNE` 最适合借鉴“本地优先 / 同步 / 客户端”的思路

---

## 3. 选型判断标准

这次分析以你当前项目的主目标为准：

- 多租户
- 知识库
- 文档管理
- 文档树
- 本地文件同步
- Web 管理台
- 后续桌面同步代理

所以看开源项目时，重点关注：

- 顶层信息架构
- 权限模型
- 文档层级模型
- 同步与导入导出能力
- 编辑器与版本策略
- 本地优先 or 在线优先
- 是否适合企业内部知识库

## 4. 重点参考项目

### 4.1 Outline

项目：

- GitHub: https://github.com/outline/outline
- 文档： https://docs.getoutline.com

我认为 Outline 是当前最值得你借鉴的“在线知识库”产品之一。

#### 值得借鉴的点

- `Workspace -> Collection -> Document` 的清晰组织方式
- 权限主要落在 `Collection` 层，而不是对每篇文档单独做复杂授权
- 支持组权限和个人权限叠加
- 支持对子树单独分享
- 支持公开发布文档和集合
- 文档修订历史和非破坏性恢复
- 导入外部 Markdown / HTML / Word / Notion / Confluence 内容

参考资料：

- Collections: https://docs.getoutline.com/s/guide/doc/collections-l9o3LD22sV
- Sharing: https://docs.getoutline.com/s/guide/doc/sharing-LG2sGOLIpl
- Revision history: https://docs.getoutline.com/s/guide/doc/revision-history-AiL6p22Ssq
- Import: https://docs.getoutline.com/s/guide/doc/import-D2ZvLqz411

#### 对 Memora 的启发

最值得直接借鉴的是这三个设计：

1. 用“知识库 / Collection”作为主要权限边界
2. 文档支持子树分享，而不是只支持单页分享
3. 历史版本恢复采用“非覆盖式恢复”，恢复本身也形成新版本

#### 不建议照搬的点

- Outline 的协作能力和整体产品已经偏成熟 SaaS，复杂度高于你当前阶段
- 它的很多细节是围绕在线协作而不是本地文件同步展开的

#### 许可证判断

- Outline 当前仓库显示为 `BSL 1.1`

结论：

- **适合借鉴设计**
- **不适合直接把代码拷进来**

---

### 4.2 Docmost

项目：

- GitHub: https://github.com/docmost/docmost
- 文档： https://docmost.com/docs

Docmost 是和你当前目标最接近的另一个参考对象，尤其适合借鉴“企业知识库协作结构”。

#### 值得借鉴的点

- `Workspace -> Space -> Page` 的层级结构很干净
- Space 作为团队 / 项目 / 部门级容器，非常适合权限收口
- 支持组权限
- 支持嵌套页面、拖拽、重排
- 支持评论、历史、搜索、附件、导入导出
- 公共分享和阅读 / 编辑模式区分清晰

参考资料：

- GitHub: https://github.com/docmost/docmost
- Introduction: https://docmost.com/docs/
- Workspace: https://docmost.com/docs/user-guide/workspace
- Spaces: https://docmost.com/docs/user-guide/spaces
- Groups: https://docmost.com/docs/user-guide/groups
- Pages: https://docmost.com/docs/user-guide/pages/

#### 对 Memora 的启发

Docmost 对你最有价值的点不是编辑器，而是“企业知识库空间设计”：

1. `Workspace` 和 `Space` 两层容器非常适合做企业组织边界
2. Space 内再承载嵌套页面，产品心智清晰
3. 组权限先于单文档权限，管理成本低

对于 Memora，我建议你可以这样吸收：

- 保留当前 `Tenant`
- 把当前 `KnowledgeBase` 视为更接近 `Space`
- 文档继续做树结构

也就是说：

`Tenant -> KnowledgeBase -> Document`

本质上就是：

`Workspace -> Space -> Page`

#### 不建议照搬的点

- Docmost 偏重在线协作页面，不强调本地目录同步
- 它的评论、协作、群组功能你现在还不应该提前全部引入

#### 许可证判断

- Docmost 仓库显示为 `AGPL-3.0`

结论：

- **非常适合借鉴模型和产品结构**
- **不建议直接复用实现代码，除非你准备接受 AGPL 约束**

---

### 4.3 BookStack

项目：

- GitHub: https://github.com/BookStackApp/BookStack
- 文档： https://www.bookstackapp.com/docs/user/content-overview

BookStack 的价值不在“协作能力”，而在“信息组织模型极其清晰”。

#### 值得借鉴的点

- `Bookshelf -> Book -> Chapter -> Page`
- 内容组织方式接近现实世界，非技术用户容易理解
- 支持内容移动和重新组织

参考资料：

- GitHub: https://github.com/BookStackApp/BookStack
- Content Overview: https://www.bookstackapp.com/docs/user/content-overview
- Organising Content: https://www.bookstackapp.com/docs/user/organising-content/

#### 对 Memora 的启发

BookStack 最大的启发是：

- 企业文档系统不一定要做得像通用块编辑器
- 有时候“层级明确、认知简单”比“无限自由”更有价值

对你的项目来说，可以借鉴两点：

1. 给知识库和目录明确命名语义
2. 强化“移动、排序、归类”这类结构操作

#### 不建议照搬的点

- BookStack 的固定四层结构太死
- 你现在需要的是可扩展文档树，不适合被固定成 shelves/books/chapters/pages

#### 许可证判断

- BookStack 仓库显示为 `MIT`

结论：

- **适合借鉴交互和内容组织方式**
- **如果确实需要，小范围实现思路也更容易合规参考**

---

### 4.4 Wiki.js

项目：

- GitHub: https://github.com/requarks/wiki
- 文档： https://docs.requarks.io

Wiki.js 的核心价值是“平台化能力”，不是单纯的页面编辑。

#### 值得借鉴的点

- 编辑器可切换
- 内容渲染链路可扩展
- 搜索引擎模块化
- 认证模块化
- 存储模块化
- 支持 Git 作为存储 / 同步后端

参考资料：

- GitHub: https://github.com/requarks/wiki
- Home: https://docs.requarks.io/s/home
- Editors: https://docs.requarks.io/s/editors
- Git storage: https://docs.requarks.io/s/en/storage/git

#### 对 Memora 的启发

Wiki.js 对你最有价值的不是 UI，而是平台结构：

1. 编辑器要和文档模型分离
2. 存储和同步不要写死成一种方式
3. 搜索、认证、存储都适合做成可替换能力层

尤其是它的 Git 存储模块很值得你借鉴方向：

- 在线知识库不一定直接把数据库当唯一真相
- 可以把“同步源”设计成独立模块

#### 不建议照搬的点

- Wiki.js 偏“Wiki 平台”，而你现在做的是“知识库 + 同步平台”
- 它的模块化深度对于你当前阶段偏重

#### 许可证判断

- Wiki.js 仓库显示为 `AGPL-3.0`

结论：

- **适合借鉴抽象层次和扩展点设计**
- **不适合直接拿代码拼进当前项目**

---

### 4.5 AppFlowy

项目：

- GitHub: https://github.com/AppFlowy-IO/AppFlowy
- Cloud: https://github.com/AppFlowy-IO/AppFlowy-Cloud
- 文档： https://docs.appflowy.io

AppFlowy 对你最重要的参考价值在于：

- 本地优先
- 桌面端 / 多端
- 同步与云服务分离

#### 值得借鉴的点

- 本地数据所有权优先
- 本地模式和云同步模式并存
- 客户端和 Cloud 分仓设计
- 自托管路径清晰

参考资料：

- GitHub: https://github.com/AppFlowy-IO/AppFlowy
- AppFlowy Cloud: https://github.com/AppFlowy-IO/AppFlowy-Cloud
- AppFlowy Cloud Docs: https://docs.appflowy.io/docs/appflowy/product/appflowy-cloud
- Self-hosting: https://docs.appflowy.io/docs/guides/appflowy/self-hosting-appflowy

#### 对 Memora 的启发

这是你当前最该认真参考的“客户端方向”：

1. 本地客户端和云服务端最好分职责
2. 本地数据不必强制依附在线系统
3. 同步是独立能力，不应耦合在普通 CRUD 里

对于 Memora，这意味着：

- `memora-web-app` 负责在线管理台
- `memora-server` 负责知识库和同步任务中心
- `memora-client` 负责本地扫描、索引、同步代理

这个方向和你当前重构方向是一致的。

#### 不建议照搬的点

- AppFlowy 已经走向大而全 workspace 平台
- 你当前不该引入它的任务、AI、数据库视图等复杂能力

#### 许可证判断

- AppFlowy 仓库显示为 `AGPL-3.0`

结论：

- **非常适合借鉴客户端 / 同步架构**
- **不适合直接把实现照搬进来**

---

### 4.6 AFFiNE

项目：

- GitHub: https://github.com/toeverything/AFFiNE

AFFiNE 更适合作为“产品方向观察对象”，而不是当前阶段的直接实现模板。

#### 值得借鉴的点

- local-first
- docs + whiteboard + table 的统一内容模型
- 强调本地优先和实时协作并存

参考资料：

- GitHub: https://github.com/toeverything/AFFiNE

#### 对 Memora 的启发

AFFiNE 能提醒你两件事：

1. 未来文档系统不一定只有线性正文
2. 文档和画布 / 附件 / 结构化块最终可能需要统一模型

但这更适合你远期，而不是当前阶段。

#### 不建议照搬的点

- 它的产品雄心过大
- 当前阶段直接模仿会把你带回“大杂烩平台”老路

#### 许可证判断

- GitHub 页显示 Community Edition 为 `MIT`
- 但项目有社区版 / 企业版区分，仍要仔细核对具体目录许可证

结论：

- **适合借鉴长期方向**
- **不适合当前阶段作为主参考**

## 5. 对 Memora 最值得吸收的设计

### 5.1 顶层信息架构

推荐综合：

- Outline 的 `Collection`
- Docmost 的 `Workspace / Space`

落到 Memora：

- `Tenant`
- `KnowledgeBase`
- `Document`

这是正确方向，应继续坚持。

### 5.2 权限模型

推荐借鉴：

- Outline 的集合级权限
- Docmost 的工作区 / 空间角色

建议：

- 权限优先落在 `KnowledgeBase`
- 文档级权限只做补充，不做主模型
- 先支持 `Owner / Admin / Editor / Viewer`
- 再支持 `Group`

### 5.3 文档结构模型

推荐借鉴：

- BookStack 的“结构优先”
- Outline / Docmost 的嵌套页面

建议：

- 保留当前树模型
- 支持目录节点和正文节点
- 强化移动、排序、重命名、路径变更
- 明确 breadcrumb 和树路径

### 5.4 版本模型

推荐借鉴：

- Outline 的非破坏式恢复
- Docmost 的页面历史

建议：

- 每次恢复产生新版本
- 版本对比支持摘要和 diff
- 后续支持命名版本点

### 5.5 导入导出模型

推荐借鉴：

- Outline 的 Markdown / HTML / Word / Notion / Confluence 导入
- Docmost 的 Markdown / HTML 导入导出
- BookStack 的结构化移动

建议：

- 第一阶段先做 Markdown / ZIP 目录导入
- 第二阶段做批量导出
- 第三阶段考虑 Confluence / Notion 迁移

### 5.6 同步模型

推荐借鉴：

- AppFlowy 的 local-first 与 cloud 分离
- Wiki.js 的 Git / Storage 抽象

建议：

- 同步做成独立任务流
- 本地客户端做扫描和变更识别
- 服务端做映射、版本、审计和发布
- 不把同步逻辑直接写进文档 CRUD

---

## 6. 不建议借鉴的方向

### 6.1 过早引入全量协作套件

比如：

- 评论
- 会议
- 任务
- 即时协作
- AI 助手

这些能力不是没价值，而是不该早于文档主链路。

### 6.2 过早引入无限块能力

像 AFFiNE / AppFlowy 这种“文档 + 数据库 + 白板 + 项目管理”的统一模型，当前不适合作为主目标。

### 6.3 直接复制受限许可证代码

尤其要注意：

- Outline：BSL
- Docmost：AGPL
- Wiki.js：AGPL
- AppFlowy：AGPL

这些项目非常适合借鉴设计，但**不要直接把核心实现拷贝进当前仓库**。

## 7. 许可证判断建议

### 7.1 可以重点借鉴思路，但谨慎代码复用

- Outline
- Docmost
- Wiki.js
- AppFlowy

### 7.2 相对更适合参考实现方式

- BookStack（MIT）
- AFFiNE Community Edition（需再确认具体目录许可）

### 7.3 最安全的策略

当前建议采用：

- 看产品
- 看数据模型
- 看模块边界
- 自己重新实现

而不是：

- 拷贝控制器
- 拷贝服务层
- 拷贝前端页面

## 9. 最终判断

如果从“最值得研究的借鉴顺序”来排，我建议是：

1. Outline
2. Docmost
3. AppFlowy
4. BookStack
5. Wiki.js
6. AFFiNE

原因不是它们谁最强，而是谁最贴近你当前阶段的目标。

对 Memora 当前最现实的借鉴组合是：

- 用 **Outline / Docmost** 定义在线知识库产品结构
- 用 **BookStack** 优化树结构和内容组织体验
- 用 **AppFlowy** 定义客户端和本地同步方向
- 用 **Wiki.js** 参考长期平台化扩展点

这条路线比“找一个项目整套模仿”更稳，也更适合你当前仓库的重构方向。
