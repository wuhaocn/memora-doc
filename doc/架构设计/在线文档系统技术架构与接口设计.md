## 6. 当前接口设计

### 6.0 当前接口成熟度说明

为了避免把“接口已存在”误读成“能力已产品化完成”，当前接口按成熟度分三层理解：

- 已成型接口：工作台、知识库列表/详情、文档树、文档编辑、版本、批量动作
- 最小闭环接口：租户隔离、租户角色控制、知识库级权限覆盖、知识库成员权限配置 API
- 骨架接口：同步触发与同步任务记录。当前只具备任务承载能力，不代表真实本地同步已经完成

因此，当前接口设计已经足够支撑继续收口在线文档主链路，但还不足以支撑继续扩展外围大模块。

### 6.1 工作台接口

#### 获取当前工作台

`GET /api/v1/workspaces/current/dashboard`

返回内容：

- 工作区信息
- 知识库统计
- 文档统计
- 最近同步任务
- 成员列表

### 6.1.1 当前会话接口

#### 登录

`POST /api/v1/auth/login`

请求要点：

- `username`
- `password`

当前说明：

- 当前仅提供 demo 登录占位
- 演示账号为 `admin / 123456`
- 登录成功后返回当前租户会话和 bearer token

#### 获取当前会话

`GET /api/v1/auth/session`

返回内容：

- 当前用户
- 当前租户
- 当前租户角色
- 当前 bearer token 占位信息

当前说明：

- 当前系统优先通过 `Authorization: Bearer demo:{tenantId}:{userId}` 解析会话
- 旧的 `X-Tenant-Id` / `X-User-Id` 仍保留兼容，用于测试和过渡
- Web 端会在登录后和应用启动后调用该接口，把本地会话补全为当前租户会话信息
- Web 端通过 `AuthContext` 与受保护路由向 `Header`、`Sidebar`、`Home`、`KnowledgeBaseDetail` 分发当前会话，避免依赖整页刷新

### 6.2 知识库接口

#### 获取知识库列表

`GET /api/v1/knowledge-bases`

#### 获取租户下知识库

`GET /api/v1/knowledge-bases/tenant/{tenantId}`

#### 获取知识库详情

`GET /api/v1/knowledge-bases/{id}`

返回补充字段：

- `currentRole`
- `canWrite`
- `canManage`
- `permissionRestricted`

#### 获取知识库成员权限列表

`GET /api/v1/knowledge-bases/{id}/members`

说明：

- 仅知识库管理角色可查看

#### 批量替换知识库成员权限

`PUT /api/v1/knowledge-bases/{id}/members`

请求要点：

- `members[].userId`
- `members[].role`

服务端约束：

- 仅知识库管理角色可配置
- 仅允许配置当前租户内有效成员
- 角色仅允许 `OWNER/ADMIN/EDITOR/VIEWER`
- 非空配置下至少保留一个知识库管理员
- 非空配置下必须保留当前操作人的知识库管理权限
- 空列表表示清空知识库级限制，回退为租户继承权限

#### 获取知识库文档树

`GET /api/v1/knowledge-bases/{id}/document-tree`

#### 获取知识库同步任务

`GET /api/v1/knowledge-bases/{id}/sync-jobs`

#### 触发知识库同步

`POST /api/v1/knowledge-bases/{id}/sync-jobs/trigger`

当前说明：

- 该接口当前只负责触发和记录同步任务
- 当前返回成功不代表已经完成真实本地目录扫描和增量同步

### 6.3 文档接口

#### 获取文档详情

`GET /api/v1/documents/{id}`

#### 创建文档 / 目录

`POST /api/v1/documents`

#### 更新文档 / 节点

`PUT /api/v1/documents/{id}`

#### 删除单个节点

`DELETE /api/v1/documents/{id}`

#### 获取知识库文档树

`GET /api/v1/documents/knowledge-base/{knowledgeBaseId}/tree`

#### 更新同级排序

`PUT /api/v1/documents/sort`

#### 批量移动节点

`POST /api/v1/documents/batch-move`

请求要点：

- `documentIds`
- `parentId`

服务端约束：

- 只允许同一知识库下节点一起移动
- 服务端自动识别顶层选中节点，避免父子节点重复移动
- 复用单节点移动时的目录成环校验

#### 批量删除节点

`POST /api/v1/documents/batch-delete`

请求要点：

- `documentIds`

服务端约束：

- 只允许同一知识库下节点一起删除
- 若选中的目录仍有未选中的有效后代，则拒绝删除
- 服务端按深度逆序删除，避免父节点先删导致约束失败

#### 获取版本列表

`GET /api/v1/documents/{id}/versions`

#### 回滚版本

`POST /api/v1/documents/{id}/rollback/{versionId}`

## 8. 当前设计决策

### 8.1 为什么先做单体服务

原因：

- 当前领域还不稳定
- 功能量还不足以支撑拆服务
- 先把模型和接口做稳更重要

### 8.2 为什么文档树采用 parentId + path

原因：

- 同时兼顾树构建和路径更新
- 更容易支持同步映射
- 后续更容易接全文路径搜索

### 8.3 为什么同步先做任务模型

原因：

- 真正的本地同步比 CRUD 复杂得多
- 需要先有记录、状态和结果承载结构
- 后续客户端接入时不需要再推翻数据层

当前判断：

- 该决策仍然成立
- 但同步模块当前仍处于“任务骨架已存在、真实链路未闭环”的阶段

### 8.4 为什么当前不继续扩 AI / 资源库

原因：

- 这些模块不会补上主链路缺口
- 会继续稀释工程重点
- 当前最缺的是文档平台基础能力

## 9. 当前技术问题

### 9.1 编译与环境问题

- `gradlew` 受当前环境网络限制，无法下载 wrapper 依赖
- 仓库自带 `gradle-8.5` 在当前机器存在 native 兼容问题
- Web 端当前机器上已通过 `npm install` 修正 `esbuild` 架构并完成 `vite build`
- 如果在跨架构机器复用已有 `node_modules`，仍可能再次出现 `esbuild` 二进制不匹配

### 9.2 代码层问题

- 当前已补 bearer token 会话占位和会话查询接口，但真实认证和租户解析尚未接入
- 已补最小租户隔离、租户成员角色控制、知识库级权限覆盖层和权限配置接口
- 已补主链路最小集成测试，并已完成当前主链路稳定执行验证
- 同步逻辑目前是占位式任务流，不是真实文件同步
- 当前编辑与版本能力已经形成基础闭环，但仍未达到高阶协作编辑水平

## 11. 当前结论

从技术上看，当前系统已经具备在线文档平台的基础骨架，但离“可生产化系统”还有三个明显缺口：

- 同步闭环
- 权限闭环
- 工程闭环

后续技术工作必须围绕这三个闭环推进，不应再回到“并行堆模块”的旧路线。
