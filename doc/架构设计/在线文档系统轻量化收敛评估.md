# Memora 在线文档系统轻量化收敛评估

## 1. 目的

本文档用于回答一个现实问题：

当前仓库已经从“多方向发散”收敛到“在线文档系统主链路”，但代码、依赖和历史模块仍有残留。为了避免后续设计和实现越来越重，需要先做一轮轻量化判断，明确：

- 什么是当前必须保留的
- 什么是当前可以冻结的
- 什么是当前应该继续减重的

这份文档强调：

`先做轻，再做全`

而不是继续在主链路之外背越来越多的历史包袱。

---

## 2. 当前主链路判断

当前真正需要保住的主链路仍然是：

`登录 -> 租户 -> 知识库 -> 文档树 -> 编辑 -> 版本 -> 权限 -> 同步任务`

围绕这条链路，当前已有实现包括：

- demo 登录占位与会话上下文
- 工作台与知识库主视图
- 知识库创建 / 编辑 / 删除
- 文档新增 / 移动 / 排序 / 批量操作
- 正文编辑与版本回滚
- 知识库级权限覆盖
- 同步任务记录与触发

结论：

- 主链路已经具备继续收口的价值
- 当前最大风险不是“功能不够多”
- 当前最大风险是“历史模块和历史依赖继续干扰主链路判断”

---

## 3. 代码与文档一致性判断

### 3.1 当前一致的部分

- 根 README、服务端 README、Web README 已统一到在线文档主线
- 核心设计文档已明确资源库 / AI / 技能系统不是当前主交付
- Web 路由当前已经收敛到登录、工作台、知识库详情三条主入口
- 后端接口已围绕会话、工作台、知识库、文档、同步任务组织

### 3.2 当前仍不够轻的部分

- 编辑器仍保留 Mermaid 图表扩展这一层额外复杂度，但已通过懒加载从主入口拆出
- `memora-client` 已收敛为最小同步代理壳，但真实目录绑定和增量同步仍未接入
- 仍缺少对旧方向彻底迁出仓库的后续收口计划

这些内容当前不一定立刻要删除，但必须明确：

- 它们不是当前产品主线
- 它们不应继续驱动架构设计
- 它们不应继续拉高当前依赖和维护成本

---

## 4. 三方依赖适配性判断

### 4.1 当前适合保留的依赖

#### Web

- `react`
- `react-router-dom`
- `axios`
- `dayjs`
- `@arco-design/web-react`
- `@tiptap/react`
- `@tiptap/starter-kit`
- `@tiptap/extension-placeholder`
- `@tiptap/extension-image`
- `mermaid`

原因：

- 这些依赖直接服务当前登录、工作台、知识库和基础编辑主链路

#### Server

- Spring Boot Web
- MyBatis Plus
- H2
- Validation
- Lombok

原因：

- 这些依赖支撑当前单体在线文档主线，复杂度可控

### 4.2 当前已识别的可减项

#### 已做的轻量化调整

- `memora-web-app/package.json` 已去掉未使用的 Tiptap 表格扩展依赖声明
- `memora-client/package.json` 已去掉当前未使用的 Arco、Tiptap、Axios、Dayjs 依赖声明
- `memora-client/package.json` 已继续去掉旧桌面 UI 不再需要的 `react-router-dom`、ESLint 和类型声明依赖
- `memora-client/src-tauri/Cargo.toml` 已去掉未接入的 `tauri-plugin-fs` 和 `tauri-plugin-path`
- `memora-server-start/build.gradle.kts` 已去掉重复声明的 `web`、`data-jdbc`、`mybatis` 依赖
- `memora-web-app` 已删除未接入主路由的 Resource / MemoraAI 旧页面、旧组件和对应 API 封装
- `memora-web-app` 已删除未接入当前编辑器交互的 `ContextMenu` 旧组件
- `memora-server` 已删除 Resource / Tag 相关控制器、服务、实体、DTO、VO、Mapper 和对应数据库表定义
- `memora-web-app` 已删除 Excalidraw 编辑器实现，图表能力收敛为 Mermaid-only
- `memora-web-app` 已将主路由、知识库详情页、富文本编辑器和 Mermaid 渲染链路改为懒加载，前端主入口包体明显下降
- `doc/功能模块` 下的旧泛设计文档已整体删除，避免与当前在线文档主线冲突
- `memora-client` 已删除旧文件管理器 / 知识库浏览类页面和对应路由壳
- `doc/架构设计/项目设计文档.md`、`doc/开发管理/迭代任务计划.md`、`doc/业界Skills`、`doc/工具使用/Skill使用及配置指南.md` 已删除

#### 当前不建议继续扩大的依赖方向

- AI 问答相关 mock 能力
- 与资源库、AI、题目生成有关的外围页面样式和组件
- 不属于同步代理方向的客户端 UI 依赖

结论：

- 当前依赖规模仍可控
- 但如果继续保留并扩写 AI / 资源库旧链路，复杂度会再次失控

---

## 5. 轻量化设计建议

### 5.1 产品层

继续坚持：

- 先把在线文档主链路做完整
- 不把资源库、AI、技能系统重新拉回首页
- 不把“以后可能会做”的能力提前写成当前架构约束

### 5.2 前端层

建议按优先级处理：

1. 保持登录、工作台、知识库详情三条主路由稳定
2. 冻结 Resource / MemoraAI 旧页面，不再继续做交互增强
3. 继续把不属于主链路的旧样式和旧工具从主应用依赖链中摘出
4. 图表能力先保持 Mermaid-only，不再重新引入重型图形编辑器

### 5.3 后端层

建议按优先级处理：

1. 继续围绕认证、权限、知识库、文档、同步任务收口
2. 不再恢复 Resource / Tag 方向的旧接口和旧模型
3. 后续如有资源能力需求，按知识库附件或文档资产子能力重新设计

### 5.4 客户端层

建议按优先级处理：

1. 保持 `memora-client` 作为同步代理最小壳，不再回退到通用桌面 UI
2. 不再给桌面端补知识库浏览和文档编辑页面
3. 在现有最小壳上继续补目录扫描、变更检测、同步上报骨架

---

## 6. 后续计划

### 6.1 P0：继续减包袱

- 修正前端依赖状态，完成 Web 端依赖声明同步
- 为前端补最小 ESLint 配置
- 修复前端 `esbuild` 架构问题
- 清理仓库中剩余的旧方向描述和无效注释
- 已完成主路由和编辑器的首轮懒加载减重

### 6.2 P1：继续收主链路

- 补登录后的统一异常反馈
- 补知识库和文档权限拒绝态
- 补创建知识库 / 创建文档 / 编辑文档的前端联调验证
- 补更稳定的会话失效处理

### 6.3 P2：继续控复杂度

- 在当前 `memora-client` 最小壳上补目录绑定和扫描链路
- 评估是否把 Mermaid 能力也降级为纯文本块扩展，继续压缩编辑器复杂度

---

## 7. 最终判断

当前最合理的策略不是“大清洗”，而是：

- 先停掉无关方向的继续扩张
- 先把依赖和文档减到与主链路一致
- 先把登录、知识库、文档、权限、同步任务做稳

一句话总结：

`当前系统可以继续做，但必须轻着做，不能再回到大而散的老路。`
